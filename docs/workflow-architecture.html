The Two Questions This Pipeline Answers

What the Data Means

Before the mechanics, the semantics. Every AP invoice document we pull is graded on two independent questions: Did we classify the document correctly? (Doc Classification) and Did we resolve the vendor to the right master record? (Vendor 2.1). But both questions rest on one underlying idea — the difference between what Auditoria produced and what the customer ended up with. That's the record lifecycle, and it's where every "Original" and "Final" value comes from.

Foundation

The Record Lifecycle: Original vs. Final Values

Every field prefixed "Original" or "Final" — and every "mismatch" the dashboard reports — traces to two source tables and the JSON blobs inside them. Understanding these is the key to the whole dataset.

Table 1 — sor.wk_inst_result (alias a): the work-instruction result

One row per processed document. It carries two JSON blobs that represent two moments in time:

Blob / ColumnMomentFields we extract
a.session Processing-time — the context captured when the document was first ingested and classified. originalRecordType → Original Record Type; messages[0] → Message ID; attachment keys/filenames → S3 locations.
a.record Live / current — the record as it stands now, after any downstream correction. recordType → Final Record Type; vendorName → vendorname (final); vendorId, invoiceNumber, dismissedComment.
a.write_status / a.active / a.isManualEntry Status flags on the row itself. Written (0/1), On UI (Dismissed/Active), Manual (Bot/Manual).

So within this one table, Original Record Type (from session) vs Final Record Type (from record) already tells you whether the document's classification changed between ingestion and now.

Table 2 — sor.wk_result_edits (alias swre): the edit diff

A row exists here only when a written record was edited. It stores a before/after snapshot as two full JSON documents — this is the literal "original value / final value" pair:

Blob / ColumnRepresentsFields we extract
swre.original_json What Auditoria AI wrote to the ERP — the machine's answer (the normalized / SOR-resolved values, not raw extractions). recordType → AAI RecordType; vendorName → OriginalVendorName (= the normalized vendor name AAI resolved to).
swre.final_json What the record became after a human edit — the customer's correction (ground truth). recordType → Customer RecordType; vendorName → Customer vendorName; aaiEntityId → joins sor.entity.
swre.wk_result_id / message_id / tenant_id Links the edit back to the result row and the edit event. FK to a.id; Edit Message ID; tenant scope.
sor.entity.entity_name Which customer entity made the edit (joined via final_json.aaiEntityId). customer entity_name.

Critical consequence: OriginalVendorName comes from swre.original_json, not from wk_inst_result. If a document was never edited, there is no edit row — so OriginalVendorName is NULL and Vendor Match Status reads "No Original Data." In the real Ochsner/GA/Coleman pull, 90% of rows had an edit record (OriginalVendorName populated); 10% did not.

The Lifecycle, End to End

1. Ingest & Classify
a.session.originalRecordType is set — the classifier's first prediction (Original Record Type).
2. Resolve Vendor & Write
AAI resolves the vendor and writes the record. That written snapshot is captured in swre.original_json → AAI RecordType + OriginalVendorName.
3. Customer Edits
A human corrects the record. The corrected snapshot lands in swre.final_json → Customer RecordType + Customer vendorName, tagged with the editing entity.
4. Live State
The edit propagates to a.record → Final Record Type + final vendorname. This is what the customer sees today.

The Two "Original vs Final" Comparisons

Because there are two source tables, there are two distinct diffs — don't conflate them:

ComparisonOriginal sideFinal sideWhat it detects
Classification drift session.originalRecordType record.recordType Record type changed between ingestion and the live record.
Edit diff (mismatch) original_json (AAI wrote) final_json (customer edited) A human overrode what AAI produced — the true accuracy signal.

Mismatch Detection Premise

The mismatch script keeps only rows where AAI's write was written and then overridden:

WHERE a.write_status = 1 -- AAI actually wrote it AND swre.tenant_id = '' AND (date range on swre) AND ( swre.original_json->>'$.recordType' != swre.final_json->>'$.recordType' -- record type edited OR swre.original_json->>'$.vendorName' != a.record->>'$.vendorName' ) -- vendor name edited

Output splits into two sheets — RecordType_Mismatch and VendorName_Mismatch (a row can be in both). Real examples from the June pull:

TypeAAI wrote (Original)Customer changed to (Final)Extracted from doc
Others → Invoice Others Invoice — (AAI under-classified a real invoice)
Vendor Onesourcehearing Starkey Hearing Technologies Starkey
Vendor Ascension Water Company Louisiana Water Company Ascension Water Company
Vendor Datex-Ohmeda GE Medical Systems Ultrasound… GE Medical Systems Ultrasound…

What the vendor examples reveal: for "Onesourcehearing," the raw ExtractedVendorName was "Starkey" — closer to the customer's correct answer than the resolved OriginalVendorName. The SOR resolution step replaced a good extraction with a wrong master mapping. In the real pull this is common: 68% of rows had Normalized ≠ Extracted (resolution altered the name), and 94% of edited rows had OriginalVendorName == NormalizedVendorName — confirming AAI writes the SOR-resolved name, so a bad hint/system-hint directly becomes a mismatch.

Concept A

Document Classification: Invoice / Others / VB_CREDIT_MEMO

When a document arrives, Auditoria's classifier assigns it a record type. For AP invoice processing there are three outcomes that matter. The reviewer's job is to confirm the classifier got it right.

Invoice
recordType = "Invoice"
A payable invoice — a vendor billing the customer for goods/services. This is the document type that flows into the AP write pipeline and gets a vendor resolved against the master SOR. The overwhelming majority of documents (~88% in production data).
VB_CREDIT_MEMO
recordType = "VB_CREDIT_MEMO"
A vendor credit memo — money owed back to the customer (returns, overbilling adjustments). It is correctly a non-invoice document. This distinction drives the single most important review exception (see below).
Others
recordType = "Others"
Anything that is not a payable invoice — statements, packing slips, remittance advices, spam, or unclassifiable pages. These should not enter the invoice write path.

Original vs. Final Record Type

Two record-type columns are pulled for every document, and the gap between them is the core signal:

ColumnMeaningSource
Original Record Type What Auditoria's model assigned when the document first arrived — the AI's own prediction, untouched. a.session → $.originalRecordType
Final Record Type What the record type is now in the live record — may reflect a customer/agent correction after the fact. a.record → $.recordType

When Original ≠ Final, the customer (or an agent) re-classified the document after we processed it — a strong indicator that our original prediction was wrong. This is exactly what the mismatch script isolates.

The Review Question & the Issue Rule

In the dashboard the reviewer answers one question — "Is an Invoice?" → Invoice or Others. From that selection, the Doc Classification Issue flag is auto-computed by comparing the reviewer's verdict against the base record type (originalRecordType, falling back to finalRecordType when the original is blank):

No Issue Reviewer says Others and the base type is VB_CREDIT_MEMO — the explicit exception. A credit memo is correctly not an invoice, so marking it "Others" is right, not a misclassification.
Issue = Yes Reviewer says Invoice but the base type is not "Invoice" — we under-classified a real invoice.
Issue = Yes Reviewer says Others but the base type is not "Others" (and not the VB_CREDIT_MEMO exception) — we mis-flagged a non-invoice as something payable.
No Issue Reviewer's verdict agrees with the base record type — the classifier was correct.

Why the VB_CREDIT_MEMO exception exists: Reviewers naturally mark a credit memo as "Others" (it isn't an invoice). Without the exception, every correctly-handled credit memo would be counted as a classification error, poisoning the accuracy metric. The rule at DocClassificationDetailsPanel.tsx encodes this: Others + VB_CREDIT_MEMO → No Issue.

Concept B

Vendor Name Resolution (Vendor 2.1)

For every invoice we must answer: which master vendor record does this document belong to? A document says "Thermo Fisher Scientific"; the customer's books call that vendor "Fisher Healthcare". Vendor 2.1 is the resolution engine that bridges the two. To audit it, we pull four different representations of the vendor name and the reason the engine landed where it did.

The Four Vendor Name Fields

FieldWhat it isSource
ExtractedVendorName The raw vendor name lifted straight off the document by the extraction model — as-printed, messy. mia.intent → extractedInfo.Fields[vendorname].Value
NormalizedVendorName The extracted name after canonicalization (casing, punctuation, legal-suffix cleanup) — the form used for matching. …Fields[vendorname].NormalizedValue
OriginalVendorName The value AAI/NLU actually wrote to the ERP at processing time — i.e. the normalized / SOR-resolved name, not the raw extraction. "Original" means "AAI's original write, before any customer edit" — the value itself is the normalized one (in the real pull it equals NormalizedVendorName ~94% of the time). swre.original_json → $.vendorName
vendorname (final) The vendor name in the live record now — reflects any customer correction. Final ≠ Original = a vendor mismatch. a.record → $.vendorName
Extract
The extraction model reads the vendor name off the PDF → ExtractedVendorName (e.g. "ACME CORP").
Normalize
Clean and canonicalize → NormalizedVendorName (e.g. "Acme Corp"). This is the value matched against the SOR sources.
Resolve against SOR
Match the normalized name against three System-of-Record sources in priority order (hints → system_hints → master). The winning source and criteria are recorded as VendorNameReason.
Write
The resolved vendor becomes OriginalVendorName. If a customer later changes it, the live vendorname diverges → flagged as a Vendor Mismatch.

The Three System-of-Record (SOR) Sources

Resolution consults three tenant-scoped tables. Each has a distinct role, and VendorNameReason tells you which one won:

sor.hints
tenant-configured rules
Human/tenant-curated matching hints: acceptable alias names, billing/supplier address patterns, currency, phone. The most authoritative source — a deterministic, configured mapping.

Key columns: vendor_name, acceptable_name, active_hints (which criteria are on), address.
sor.system_hints
system-learned mappings
Mappings the system learned automatically from past write history — "last time we saw this extracted name, it resolved to this vendor." Captures extracted_name → initial_vendor_name → final_vendor_name chains.

Key columns: extracted_name, initial_vendor_name, final_vendor_name, write_status.
sor.master_sor
the canonical vendor registry
The customer's master vendor list — the source of truth for who a vendor is. Every resolved invoice should point at an active row here. "Exists in mast_sor" in the review form asks exactly this.

Key columns: coalesced_name (canonical vendor name), status (active/inactive), address1/address2.

VendorNameReason — the Resolution Verdict

This single field encodes how the vendor was resolved. Its prefix names the winning source; the suffix names the criteria and match strength. Real values from production data:

Reason prefixMeaningSOR tables consulted
fromSystemHints Resolved from a system-learned mapping. Most common in production (~46%). ::ambiguousEntity suffix = multiple candidates existed. sor.system_hints + sor.master_sor
withLLM No deterministic hint matched; an LLM resolved the vendor against the master registry (~22%). sor.master_sor
withHints::…::exactMatch Matched a configured hint. The middle segment lists the criteria that fired — e.g. acceptableNames+billContains+billingAddress+supplierAddress. sor.hints + sor.master_sor
withHints::acceptableNames::legacyExactName Matched an acceptable-name alias via the legacy exact-name path. sor.hints + sor.master_sor
noMatchWithHintsLLM Neither hints nor LLM could resolve the vendor — needs manual attention. —

Why the dashboard re-matches locally: The reason tells you the engine's verdict, but not which specific SOR rows support it. So post-processing takes the Extracted and Normalized names and re-runs the match against each tenant's sor.hints, sor.master_sor, and sor.system_hints tables — populating six columns with the top-10 matching rows. The reviewer sees the actual candidate records side-by-side with the document, and can confirm whether the chosen vendor is right.

The Vendor 2.1 Review Verdict

After inspecting the four names, the reason, and the SOR candidates, the reviewer records:

FieldValuesMeaning
Vendor 2.1 Matching Issue Does not Exist / Vendor Matching Issue / (none) "Does not Exist" = vendor isn't in master SOR yet; "Vendor Matching Issue" = matched to the wrong vendor.
Exists in mast_sor Yes / No Whether the resolved vendor has an active row in sor.master_sor.
Expected Vendor Name free text The correct vendor name, when the reviewer disagrees with the resolution.

These vendor fields are disabled when "Is an Invoice" = Others — vendor resolution is only meaningful for actual invoices.

How the Data Gets Here

The Pipeline

With the semantics established, here is the machinery that produces those fields — from the Metabase pull through processing, the dashboard, review, and export.

Phase 1

Data Pull from Metabase

The extraction script ap_invoice_data.py connects to two Metabase instances (Regular + Enterprise) and runs tenant-isolated SQL against the SOR database shards. A separate script ap_invoice_mismatch_data.py pulls records where the customer changed AAI's original classification or vendor name.

Architecture

InstanceURLPurposeDB Selection
Regular metabase.auditoria.ai Primary SOR databases Auto-filter for names containing sor or shard
Enterprise metabase-ent1.auditoria.ai Enterprise tenant shards DB 6 prioritized; DB 18 force-appended

Two-Step Tenant Discovery

  1. DB Discovery (Probe): For each tenant ID, a fast COUNT(*) query runs against every database on every instance, filtering by tenant.tenant.id within the configured date range. The first database returning count > 0 is cached as that tenant's DB location.
  2. Per-Tenant Fetch: The full SQL template runs once per tenant against its discovered DB. Date range and tenant ID are injected via string replacement of DATE_FROM_PLACEHOLDER, DATE_TO_PLACEHOLDER, and TENANT_PLACEHOLDER.

The main query joins five tables to assemble each document's full context:

Async Query Handling

Metabase may return HTTP 202 for long-running queries. The script polls /api/dataset/query_key/{qk} every 5 seconds for up to 60 attempts (5 minutes). If a via[].status == 'failed' is detected, it aborts early.

Mismatch Script Differences

ap_invoice_mismatch_data.py drives from sor.wk_result_edits instead of sor.wk_inst_result and applies an additional WHERE filter:

WHERE a.write_status = 1 -- only Written records AND (swre.original_json->>'$.recordType' != swre.final_json->>'$.recordType' OR swre.original_json->>'$.vendorName' != a.record->>'$.vendorName')

Output splits into two Excel sheets: RecordType_Mismatch and VendorName_Mismatch, with extra columns for Edit Message ID, customer entity_name, AAI RecordType, and Customer RecordType/VendorName.

Phase 2

Post-Processing Pipeline

After combining all tenant DataFrames (deduped by Document ID, sorted by created_at DESC), four transformations run in sequence:

1. Vendor Name JSON Update
Parses vendorname_json for each row. If any entry's Value or NormalizedValue is a case-insensitive substring of OriginalVendorName, it overwrites ExtractedVendorName and NormalizedVendorName with the matched entry's values.
2. Date Formatting
Converts created_at and updated_at to human-readable format: "July 08, 2026, 3:45 PM".
3. SOR Matching (Bulk Fetch + Local Match)
The heaviest step. For each tenant, determines which sor.* tables are needed based on VendorNameReason values in the data, bulk-fetches those tables once, then matches every row locally using a Python reimplementation of MySQL's LIKE '%value%'. Adds 6 new JSON columns with top-10 matches from sor.hints, sor.master_sor, and sor.system_hints.
4. JSON Date Formatting
For the 6 SOR matching columns, parses embedded JSON arrays and reformats any created_at/updated_at keys within objects to the human-readable format.

SOR Matching Logic by VendorNameReason

ReasonTables FetchedMatch Targets
withHints sor.hints + sor.master_sor Match against hints._match_data (full JSON) and master_sor.coalesced_name
withLLM sor.master_sor Match against master_sor.coalesced_name only
fromSystemHints sor.system_hints + sor.master_sor Match across extracted_name, initial_vendor_name, final_vendor_name in system_hints; plus master_sor.coalesced_name

Excel Output Format

The final DataFrame (37 original + 6 SOR columns = 43 columns) is written to an .xlsx file using openpyxl. ID columns (Document ID, Tenant ID, Message ID, vendorid) are cast to integer strings with text formatting ('@') to prevent Excel scientific notation. If total rows exceed 5,000, data splits across multiple sheets (Data_Part_1, Data_Part_2, etc.).

Phase 3

Excel Upload Flow

  1. User drops .xlsx file(s) into the ExcelUploader modal (supports drag-and-drop, multiple files).
  2. All sheets are read; column names are normalized case-insensitively against the known 37-column schema.
  3. Each row is parsed through parseDocClassificationRow() — dates, JSON columns, Yes/No fields, and SOR columns are all typed.
  4. If a Reviewed column exists (resume file), reviewed documents and their review data are extracted.
  5. Auto-suggestion (not auto-save): Eligible documents — finalRecordType == 'Invoice' AND originalRecordType == 'Invoice' AND originalVendorName matches a name in any SOR column — get their review form pre-filled (isAnInvoice: 'Invoice', vendor21MatchingIssue: 'Does not Exist', existsInMastSor: 'Yes') and a banner appears. Nothing is saved until the reviewer clicks Mark & Next — so every reviewed document is human-confirmed, and accuracy metrics reflect real human judgment.
  6. Parsed documents are held in AppContext state; reviews persist to browser localStorage as you work.

PDF-Document Matching

When a ZIP of PDFs is uploaded, documents are matched by comparing the UUID extracted from the PDF filename (e.g., 924b0482-f146-11f0-b3cd-b7a39ee49af9.pdf) against the last path segment of each document's S3 location. Only matched documents appear in the review queue when PDFs are loaded.

Review Actions

  1. Mark & Next: Saves the review (isAnInvoice, expectedDocType, vendor21MatchingIssue, existsInMastSor, expectedVendorName, comments) and advances to the next unreviewed document. Doc Classification Issue is auto-calculated by comparing isAnInvoice against originalRecordType.
  2. Skip: Hides the document from the default review queue (session-level tracking, not persisted).
  3. Auto-fill: When vendorMatchStatus is "match", originalRecordType is "Invoice", and record types match, a one-click button fills "Invoice + Does not Exist + Yes".
  4. Update & Next: Available for already-reviewed documents when revisiting via the Reviewed sheet.

Special Case: VB_CREDIT_MEMO marked as "Others" is NOT flagged as a Doc Classification Issue — this is an intentional exemption in the auto-calculation logic.

Persistence Strategy

Reviews are written to browser localStorage as you go — immediate and offline-safe, so progress survives a refresh. The uploaded dataset and PDFs live in memory for the session. Nothing leaves the browser; export is an explicit action.

Embedded JSON Structures

Three columns carry nested JSON. Their real shapes:

doc_classification_json — per-page classifier output
[{ "start_page": 1, "end_page": 1, "prediction": { "model": "LLM", // or "LSTM" "classification": "Invoice", // the predicted record type "confidence": 0.96, "languages": { "English": 0.99 }, "raw_text": "…OCR text…", "page1_text": "…full page text…", "Reason": "" } }]
vendorname_json — extraction field object
[{ "Name": "vendorName", "Type": "Name", "Value": "Fisher Healthcare", // → ExtractedVendorName "NormalizedValue": "Fisher Healthcare", // → NormalizedVendorName "Reason": "withHints::acceptableNames+supplierAddress::exactMatch", "Confidence": 0.9, "FuzzyMatchEnabled": false, "aaiNamespace": "invoice" }]
SOR lookup rows — hints / master / system_hints (differing keys)
// sor.hints → vendor_name, acceptable_name, active_hints, address // sor.master_sor → coalesced_name, status, address1, address2 // sor.system_hints → extracted_name, initial_vendor_name, final_vendor_name, write_status // common to all → search_value_used, tenant_id, created_at, updated_at

Auto-suggestion eligibility: a document's form is pre-filled only when both record types are "Invoice" and a SOR name field (coalesced_name / vendor_name / acceptable_name) matches OriginalVendorName. This is a suggestion — it never saves on its own; the reviewer confirms with Mark & Next. So nothing is dispositioned automatically and accuracy reflects only human judgment.

Data-quality caveat: doc_classification_json is pulled via CAST(… AS CHAR(10000)) in SQL. On documents with long OCR text the JSON is truncated mid-string and becomes unparseable. The dashboard's parseJsonColumn catches this and keeps the raw string while setting the parsed value to null — so the JSON Data tab shows nothing structured for those rows. The other JSON columns (vendorname_json, SOR columns) are not affected.
